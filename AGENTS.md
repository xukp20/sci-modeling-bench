# SciModelingBench 项目协作说明

## 1. 项目定位

- 本仓库用于实现 SciModelingBench 的公开框架、数据接入和评测能力。
- 当前仍处于架构讨论阶段。没有明确决策前，不提前冻结 Dataset、Objective、Protocol、Task 等公开接口。
- 优先实现当前已经确认的最小需求，不为尚未确定的实验方式增加抽象层或扩展点。

## 2. 顶层目录

- `src/`：可发布的源代码。业务边界确定后再创建包内子目录，不预先复制尚未落地的架构图。
- `tests/`：自动化测试。后续按照真实代码边界划分单元测试和集成测试。
- `docs/`：可复用、可共享、可提交的长期文档。
- `dev_docs/`：本地开发过程文档，默认被 Git 忽略，不上传到公开仓库。
- `data/`：本地数据缓存，默认被 Git 忽略。公开数据应由独立数据仓库托管，代码仓库只保存 manifest、校验信息和加载逻辑。
- `artifacts/`：本地生成结果、临时模型和评测产物，默认被 Git 忽略。

## 3. 文档语言

- `README.md` 和 `docs/` 下的公开文档默认使用英文。
- `AGENTS.md` 和 `dev_docs/` 下的本地开发文档默认使用中文，可保留必要英文术语。
- 引用论文、数据集、代码仓库或网页时，应记录标题、URL、访问日期以及版本或 commit（如果可用）。
- 不确定的接口、数据行为和实验结论必须标记为“待确认”，不得根据计划推断成已经实现的事实。

## 4. `dev_docs/` 主题划分

本地开发文档按实际主题维护，避免把所有内容写入单一长文档：

- `dev_docs/design/`：项目边界、术语、架构选择和设计决策。
- `dev_docs/implementation/`：已经确认范围的实现计划、迁移记录和审计结果。
- `dev_docs/reference_research/`：外部 benchmark、数据源、依赖库和相关项目调研。
- `dev_docs/code_implementation_reference/`：当前已经落地的代码结构、公开接口和调用关系；不得提前描述尚未实现的设计。
- `dev_docs/testing/`：测试环境、数据校验方法、复现命令和已知限制。
- `dev_docs/dev_records/`：按日期记录开发过程、验证结果和未决问题。
- `dev_docs/working_on.md`：当前正在推进的任务和状态。

新增稳定主题时可以增加语义明确的子目录，但不要使用 `phase1`、`misc`、`temp` 等无法长期表达内容范围的名称。

## 5. 文档命名

- 每日记录使用 `YYYY-MM-DD.md`。
- 带明确日期的设计或调研记录使用 `YYYY-MM-DD_<主题>.md`。
- 长期维护的主题文档使用可读的业务名称，例如 `<数据集名称>_数据来源与许可.md`。
- 每个包含多份文档的主题目录应维护 `README.md`，说明范围并提供索引。
- 文件名应表达问题本身，不使用 `final`、`new`、`latest` 等容易失效的状态词。

## 6. 开发与变更约束

- 修改前先阅读相关 `docs/`、`dev_docs/` 和现有代码，避免建立重复概念。
- 只修改完成当前任务所需的文件，不顺带重构无关内容。
- 新增公开抽象前，应先说明它解决的当前重复或边界问题；仅为可能的未来需求服务时不要添加。
- 数据加载与转换必须保留来源、版本、许可证、校验和以及 raw-to-canonical provenance。
- 不提交原始大文件、模型权重、真实凭据、本地缓存或来源不允许再分发的数据。
- 实现变更后运行与改动范围匹配的最小测试，并在无法验证时记录剩余风险。

## 7. Git 约定

- `dev_docs/` 是本地过程记录，不使用 `git add -f` 强制提交，除非用户明确要求公开某份文档。
- 提交前检查暂存区，确认没有数据文件、模型、缓存、密钥或其他本地产物。
- Commit message 默认使用 Conventional Commits，例如 `feat(core): ...`、`docs: ...`、`test(data): ...`。
- 不覆盖或回退用户及其他任务留下的未提交改动。
